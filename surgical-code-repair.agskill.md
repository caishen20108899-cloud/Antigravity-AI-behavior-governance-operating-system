# 🪄 Antigravity 专精能力卡：外科手术式代码修复与架构变更

> 这是 Antigravity 的渐进式技能卡 v2.0。当你接到需要打开源码进行重构、改写或大面积修复的指令时，应当挂载此卡。

## 🎯 挂载与触发 (When to Mount)

- 当用户需要你在现成的代码仓中新增一个功能时。
- 当发生代码级别的报错，需要你溯源修改逻辑时。
- 当你需要执行重构与提取变量操作时。

## 🛡️ 逻辑沙盒执行 (Dry-Run Sandbox Gate)

> 在决定动刀子改 `xxx.py` / `xxx.js` 前，强制自我审查以下红线。

- **外科手术断言 1**：本次修改影响多少个函数？是否能够**绝不修改与之无关**的其他注释或方法？
- **外科手术断言 2**：用户要我修改的是问题 A，有没有可能因为我的改动导致文件里其他被依赖的方法崩塌？
- **简单至上断言 3**：我准备发出的新代码，是不是过度工程化了？有更土、但最安全有效的方法吗？

## ⚙️ 标准执行流 (Procedure)

> 将每一次的大规模开发拆解成 Karpathy 倡导的“闭环验证驱动 (Goal-Driven Execution)”。

1. **先出计划再动刀 (Explicit assumption)**
   - 在向文件下发 `replace_file_content` 工具前，必须用纯中文罗列“改动点范围在哪”，明确对用户交代绝不乱动其他逻辑。
2. **极简注入与定点切除 (Surgical Fix)**
   - 寻找替换切片时，首行和尾行紧紧贴住需要改动的函数，**不要跨函数覆盖**！
   - 原文里写得丑的逻辑和冗余的代码，除非用户点名要求打扫，否则权当没看见。
3. **闭环效验 (Verify Goal)**
   - 修改完成，**绝不“提交即走”**。这也并称为「防盲写铁律」(Anti-Blind Coding) 的最后一道门禁！
   - 此操作要求立刻用 `run_command` 跑编译命令（`make` / `npm build` / `python -m syntax` 等检查）。如果你所在环境因为隔离无法验证，必须生成对应脚本让 `ag_remote_verify` 替你去测！
   - 让改动结果立刻跑通测试循环产生 `Exit Code 0`，方能交差挂起 ✅。没有物理日志结案一律被判定为严重的欺骗执行！
