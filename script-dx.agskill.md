# 🚀 Playbook: 脚本与工具 DX (Developer Experience) 行动基准

> 根据历史故障（如跨平台向 Windows 分发 `tailscale` 部署脚本时的报错连天），我们对运维与打包产物的输出有了更高阶的底线要求。任何被 AI 生成用于给最终用户/开发者执行的代码，必须接受本框架的拷问。

## ⏱️ 金标准：TTHW (Time-To-Hello-World)
我们采用 TTHW 衡量你的脚本质量。
- **Champion 赢家标准**： `< 2 min`。用户只要敲下 `./your_script.sh`，不用回答几百个交互问题，不用手动帮脚本下依赖库，它就能立刻跑通并输出 `SUCCESS`。
- **Red Flag 极度危险**： `> 5 min`。如果脚本要求用户去查环境变量怎么配，去手动寻找被占用的端口并强杀，那么这是一坨答辩代码，你必须将这类烂工作自动化掉。

## 🛡️ 生成跨端脚手架的 4 项天条 (Zero Friction Rules)

当你在向外界（特别是 Windows 用户）生成环境安装或启动 `.bat`/`.sh` 时，你不可触及的红线：
1. **幂等性 (Idempotency)**：必须！不管用户运行你的脚本 1 次还是 50 次，不会产生重复的文件层叠，也不会因为二次运行而抛出 `File exists` 甚至损坏系统。
2. **渐进式排雷 (Fight Uncertainty)**：任何涉及网络下载、拉取或连接服务（例如代理接通）的操作前，先发一个 `ping` / 探针看有没有通，没通就给出友好的中文红字提示（如：`🔴无法连接云端，请检查是否处于内网环境`），而不是生硬地抛出 `curl: timeout exit(52)`。
3. **消除盲盒感 (Colorful Logging)**：不能让用户盯着一个一直在转且没有输出的控制台。只要步骤耗时 > 3 秒，你必须在之前打上一句 `Echo "正在配置网络核心驱动，请稍候约 10 秒..."`。
4. **清理现场 (Cleanup)**：如果脚本构建或下载中途出错 `Crash`，你的代码应当利用类似于 `trap` (bash) 或者 `try-catch/finally` 的机制清理残骸临时文件，绝不做给用户的系统拉屎不管擦屁股的流氓脚本。

## ⚔️ 执行审核口令
写完脚本后，默念三遍并检查：
- `这是给没耐心的人运行的。`
- `他们不懂这背后的原理，只有红绿色的结果能让他们安心。`
- `我不允许它在未预期的问题下吐出恶心的系统底层调用栈。`

## ⚖️ 多兵种协作与质量门禁 (Quality Gate & Veto Protocol)
从古老的“门下省驳回制”汲取灵感，现针对跨兵种的交互规定以下必须强制进行的阶段性审查：
1. **防幻觉门禁 (Veto Hub)**：`@Backend` 或 `@Frontend` 的每一次跨核心工程变动（大范围重构、跨模块调用），在投产之前必须主动在看板向 `@Architect` （或 `@DevOps` / `@Security`）发起【查验请求】。如果查验不通过，`@Architect` 应当直接打上 `[Veto 封驳]` 标签并将任务退回原驻列以防污染主仓库。
2. **状态机强制隔离 (Immutable Stages)**：任务仅能在新版的五段式架构（待规划 -> 评审 -> 编码 -> QA -> 结案）中单向平滑移动。绝对禁止越过「QA 测试」这道墙壁直接把没有本地验证的代码宣发给长官汇报完成。
3. **静默工作流 (Context Shielding)**：代码的报错调用栈绝不允许全盘倾倒在全局通讯公频中，必须由产生 bug 的当前执行者利用工具自我化解并在日志中留存，仅将**差值（Diff）和极简摘要**反馈给评审者，保持战术信道的畅通无阻。
