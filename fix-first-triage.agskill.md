# ⚡ Fix-First 三分类审查法

> 核心思想：不是所有发现都需要问用户。机械性问题直接修，需要判断的批量问，建议性的仅记录。

---

## 三分类定义

### [AUTO-FIX] 直接修复 — 不询问

满足以下**全部条件**的发现，直接修复：

- ✅ 修复方式只有一种（没有歧义）
- ✅ 修改后不会改变业务行为
- ✅ 修改少于 5 行
- ✅ 属于以下类别之一：

| 类别 | 示例 |
|------|------|
| 格式/排版 | 多余空行、缩进不一致、行尾空格 |
| 拼写错误 | 变量名/注释中的错别字 |
| 未使用的 import | 导入了但未使用的模块 |
| 明显的 null 守卫 | `obj.property` 缺少 `obj?.property` |
| console.log 遗留 | 调试日志未清理 |
| CSS 机械修复 | `outline: none` 缺少替代焦点样式 |

**执行后汇报格式**：
```
[AUTO-FIXED] path/to/file.ts:42 — 移除未使用的 import
[AUTO-FIXED] path/to/file.ts:88 — 修复拼写 'recieve' → 'receive'
```

---

### [ASK] 批量询问 — 累积后一次提问

满足以下**任一条件**的发现，归入 ASK 队列：

- ⚠️ 修复方式有多种选择
- ⚠️ 修改可能改变业务行为
- ⚠️ 涉及架构决策
- ⚠️ 不确定是否是「刻意设计」

**累积规则**：
1. 发现问题时不立即询问，先标记为 ASK
2. 继续审查/修复其他 AUTO-FIX 问题
3. 所有 AUTO-FIX 完成后，将所有 ASK 项**批量合并**为一次提问

**批量提问格式**：
```
🔍 以下 {N} 个发现需要你确认方向：

1. path/to/auth.ts:42 — 密码哈希用的 MD5，是否升级为 bcrypt？
   推荐: A) 升级为 bcrypt（推荐）B) 保持现状 C) 跳过

2. path/to/api.ts:100 — 未处理 404 响应，是否添加错误处理？
   推荐: A) 添加统一错误处理（推荐）B) 仅添加日志 C) 跳过

3. path/to/config.ts:15 — API 密钥硬编码，是否迁移到环境变量？
   推荐: A) 迁移（推荐）B) 跳过

请回复选项，如：1A 2A 3B
```

---

### [INFO] 仅记录 — 不行动

满足以下**任一条件**的发现：

- 💡 是优化建议而非问题
- 💡 严重性低，不影响功能
- 💡 属于「最佳实践」而非「必须修复」
- 💡 不确定性高（置信度 < 5/10）

**记录格式**：
```
[INFO] path/to/file.ts:42 — 建议：可考虑使用 Map 替代 Object 以提升查找性能
[INFO] path/to/styles.css:100 — 提示：此组件的 z-index 值(999)偏高，可能在复杂布局中产生层叠冲突
```

---

## 置信度门控

每个发现必须附带置信度评分 (1-10)：

| 置信度 | 含义 | 处理规则 |
|--------|------|---------|
| 9-10 | 阅读代码验证过，确认是 bug | 按类别处理 (AUTO-FIX/ASK) |
| 7-8 | 高度匹配已知模式，极大概率正确 | 按类别处理 |
| 5-6 | 中等，可能是误报 | 降级为 ASK（即使本来是 AUTO-FIX） |
| 3-4 | 低置信度 | 降级为 INFO |
| 1-2 | 猜测 | 不报告（除非严重性为 P0） |

## 与现有体系集成

### 在 surgical-code-repair 中使用

在 Step 2.3 的「分类」步骤中，按本技能卡的三分类法执行。

### 在多智能体协作中使用

当 `@Backend` 或 `@Frontend` 完成代码修改后，在移交给 `@DevOps` 验收前，
必须先对自己的修改执行 Fix-First 三分类自检。

### 在全局规则中引用

当系统规则要求「修后三验」时，Fix-First 的 AUTO-FIX 日志可作为第一层验证的证据。

---

## 校准学习

> 来自 gstack confidence.ts 的自适应校准机制

如果你报告了一个置信度 < 7 的发现，且用户确认它**确实是**问题：
→ 这是一个**校准事件**——你的初始置信度偏低
→ 记录修正后的模式到 `_learnings.jsonl`，使未来审查以更高置信度捕获此类问题

如果你报告了一个置信度 > 7 的发现，且用户说**这不是问题**：
→ 这也是一个**校准事件**——你的初始置信度偏高
→ 同样记录，避免未来误报
